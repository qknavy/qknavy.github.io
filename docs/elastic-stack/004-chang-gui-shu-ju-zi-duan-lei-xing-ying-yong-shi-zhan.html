# 1.2、Mapping映射

ES在存储数据的时候会保存几个部分：元信息、原始数据（_source）等

其中原始数据可以自己设置是否开启（默认开启），不开启的话不会将原始数据保存到es，而只会保留一些元信息及索引信息（id等）

如果我们不想将原始数据放到es，可以在创建索引mapping的时候禁用_source，这样就可以保存更多的数据：

PUT qknavy-company-001
{
  "mappings": {
    "_source": {
      "enabled": false
    }, 
    "properties": {
      "companyName": {
        "type": "text"
      },
      "createDate": {
        "type": "date"
      },
      "city": {
        "type": "text"
      }
    }
  }
}

# 2、常用字段类型应用

官网介绍 (opens new window)

# 2.1、string字符串

  • text：文本

    按规则分词，至少1个次，默认分词器standard

    应用场景：用于需要分词检索的场景

  • keyword：关键词

    关键词类型，不分词，仅仅一个词，不会分词，6.x之前通过string。不能指定analyzer

    应用场景：固定文本信息，无需全文检索的场景。比如姓名、省份、商品类目

可以通过_analyzeapi去查看两者之间的区别

POST _analyze
{
  "analyzer": "standard",
  "text": ["beijing union university"]
}

# 4、字段类型自动检测（field_detect）运用

在动态mapping的情况下，插入数据到es，es会自动推断并进行映射。es默认启动。

字段自动推断类型：

  • 日期类型自动推断

    必须符合一定的格式，否则会出错

  • 数值类型自动推断

    所有类型都推测为long类型

注意：在没有充分把握的时候不要主动关闭，保持默认即可

# 5、字段模型设计

# 5.1、单字段与多字段

如系统版本号，有时候需要进行比较。默认情况下使用一个字段保存，但是在这种场景下建议采用3个字段，分别保存大版本、小版本等

# 5.2、text和keyword

text支持全文检索，支持分词

keyword不支持分词。如果保存的是一个不可分的词不需要分词的场景的话建议优先使用keyword

# 5.3、数值与词项

如果数值类型需要进行范围查找的话可以定义为数值类型，但是如果不需要范围查找，只需要精确查找的话建议优先采用词项keyword类型（比如人的性别0或1这种场景）。因为它是基于key-value的形式，倒排索引的搜索效率比数值类型要高。

# 5.4、整型浮点

如果有精确的汇总统计的时候建议采用整型而不要用浮点型。如果仅仅是检索的时候可以采用浮点型（或者扩展浮点scale_float）

# 5.5、数值与范围

数值与范围本质上都一样，因为它的底层是实现是一样的，但是范围类型对于查询会更加方便一些。

# 6、索引字段类型设计限制

# 6.1、字段的数量限制

单个文档最大的字段数量：官网默认1000，可以修改（不建议）

# 6.2、对象字段深度限制

默认20，不建议太深。一般使用最多三层，因为层级太深会直接影响性能

# 6.3、字段命名规范

尽量英文名、驼峰
