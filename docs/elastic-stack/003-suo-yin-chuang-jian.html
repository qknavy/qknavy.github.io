3、索引创建规范约束及数据模型

Navy 2020-08-22

ES是文档型数据库，不是关系型数据库，不具备严格的ACID事务特性，所以，不要企图在严格事务性场景中运用它

# 1、索引创建方式与规范

  • 索引概念：

  • 索引别名

    创建索引的时候默认会创建一个和索引同名的别名。别名对于数据迁移非常有用，只需要将别名指定到另外一个索引即可，不像关系型数据库需要建表、迁移数据、删表等步骤

  • 索引命名

为了管理的便利性，需要一定的命名规范：

  • 业务类型：索引名+数字版本

如：商品索引product_001，店铺索引store_001

  • 日志类型：索引名+日期时间

如系统日志system-log-2020.08.21.001，业务日志product-log-2020.08.21.001

  • 应用意义

数据隔离、 查询便利、 模糊匹配、 精确指定索引

# 1.2、索引必备的设置与默认配置

# * 静态设置

索引分片：默认为1，创建索引时需要指定，之后索引活动期间不可变，后期只能重建

副本和刷新时间可以调整

索引字段类型：字段类型不可变，一旦指定几乎不可变。若要修改，则需要刷新所有历史数据，等同于索引重建

# 2、索引文档的数据模型

数据模型类别（对比传统关系型数据库）：

  • 数据库=ES集群（一个关系型数据库等同于一个ES集群）

  • 数据表=ES索引（一个数据表等同一个ES索引。新版本不支持一个索引多个type。）

  • 行记录=ES文档（一条数据等同一条ES文档）

索引的字段名不要有点.，可以下划线中划线等都可以，这是由于es数据底层存储结构导致的。

# 2.1、文档数据模型介绍

  • 文档类型：

json类型

  • 元数据：

_index：索引名称

_type：默认值__doc新版本固定下来

_id：数据的唯一标识，没有指定会自动生成

_version：数据的版本

_seqno：插入数据更新数据的顺序号，自增

_primary_term：主分片的顺序号，每次主从分片切换会增加，类似于epoch

其它（如查询的时候会返回一个found）

  • 源数据

json结构

# 3、索引设计经验

  • 索引与分片与副本关系

分片数量： 索引分片数不超过节点数量；1个索引40个分片等同于40个1分片索引

副本数量：

索引副本数量少于节点数量；副本数量可多可少。

  • 分片容量

数据容量：

分片数据容量上限不超过50GB，建议20GB-40GB

数据条数：

单分片不超过2的32次方-1（21亿条）

  • 数据模型设计原则

简单模型原则

能用平铺的模型搞定的，尽量平铺；对象深度建议不超过2级。

最细粒度原则

对象嵌套遵循最细粒度原则，上层数据冗余处理。

多索引原则

单一索引尽量满足单一业务场景；不同的业务场景索引及时数据模型相似也要分开。

大宽表索引原则

特殊场景合并多个表时，尽量全部合并在一起；ES不支持join，大宽表课解决查询问题。
