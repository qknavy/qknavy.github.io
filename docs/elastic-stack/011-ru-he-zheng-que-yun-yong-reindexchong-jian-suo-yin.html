11、如何正确运用reindex重建索引

Navy 2020-09-02

ElasticSearch是文档型数据库，不是关系型数据库，不具备事务特性，所以千万不要运用在严格事务性场景的应用项目中

# 1、重建索引应用场景

# 1.1、集群版本升级

版本升级的时候，由于版本之间存在一定的兼容性（特别是大版本之间会存在索引的读写兼容性问题），需要重建索引

  • 同一个大版本原则上可以直接升级
  • 跨大版本存在索引不可用的可能性

# 1.2、索引远程迁移

集群迁移，迁移索引服务不停机，数据提前迁移

集群A --> (迁移)--> 集群B

# 1.3、索引分片数量调整

当原有的分片数量太小，需要手动调整的时候，需要用到索引重建

  • 索引分片数量太多
  • 索引分片数量太少

# 1.4、索引文档结构变更

  • 文档字段类型变更

    比如将原来的text类型改为keyword类型，修改的mapping对原有的数据不生效，查询的时候可能会出故障

  • 字段属性变更

    比如之前的数据不需要被检索，而新的数据需要走索引，这时需要开启index=true，但是这样一来如果查询的时候老的数据可能就会出错

  • 文档对象结构变更

    比如原先是flatter扁平结构，新的数据需要整合在一起组成一个对象或者数组

# 1.5、索引碎片垃圾

Lucene问题，它在大量的增删改查之后会存在一定的垃圾，如果垃圾比较多的话会占用资源导致es性能下降

# 2、常用重建索引的方式

# 2.1、重建索引解读（Reindex）

重建是创建新的索引，原有索引保留。

重建索引的时候，原有索引的_source必须开启，否则会找不到原始数据

# 2.2、重建索引url请求参数解读

URL参数：

  • refresh

    目标索引是否立即刷新设置。重建索引相当于不断地往新的索引中插入数据

  • wait_for_active_shards

    重建索引分片响应设置

  • scroll

    快照查询时间

  • slices

    重建并行任务切片数

  • Max_docs

    单次最大数据量，条数

  • Requests_per_second

    每秒单次执行的重建文档数

# 2.3、重建索引req请求参数解读

请求参数：

  • conflicts

    索引数据冲突如何解决，直接覆盖还是中断【目前固定值：proceed，表示覆盖】

  • source{xx}

    源索引配置信息

  • desc{xx}

    目标索引配置信息

  • script

    脚本处理，修改原索引数据信息之后写入到新的索引

# 2.4、rep响应参数

响应参数：

  • 成功数
  • 更新数
  • 新增数
  • 失败数

# 2.5.5、路由重建（routing）

首先创建索引，设置3个分片（分片数一般和自己的es集群节点数一致）

PUT kibana_sample_data_flights_reindex
{
  "settings": {
    "number_of_shards": 3
  }
}

重建索引

POST _reindex
{
  "conflicts": "proceed", 
  "source": {
    "index": "kibana_sample_data_flights"
  },
  "dest": {
    "index": "kibana_sample_data_flights_reindex",
    "routing": "=flight "
  }
}

routing即为要路由方式，取值范围：不设置(unset)、"keep"、"discard"以及"=xxx"

GET kibana_sample_data_flights_reindex/_search?track_total_hits=true&routing=123

# 3.1、条件式重建索引

POST _reindex

  "source": {
    "index": "kibana_sample_data_flights",
    "query": {
      "match": {
        "DestAirportID": "MAN"
      }
    }
  },
  "dest": {
    "index": "kibana_sample_data_flights_reindex"
  }
}

上述操作将kibana_sample_data_flights索引的所有DestAirportID字段为MAN的数据导入到索引kibana_sample_data_flights_reindex下

这种方式可以用来做日志归档，比如每个月执行一次，将上个月的数据写入到一个新的索引（作为历史数据）

但是上面查询方式重建索引可能查询到的数据比较多，写入新索引的数据就比较多，有时候只需要这些数据的一部分，就需要控制导入的新数据条数（max_docs）：

POST _reindex
{
  "max_docs": 10,
  "source": {
    "index": "kibana_sample_data_flights",
    "query": {
      "match": {
        "DestAirportID": "MAN"
      }
    }
  },
  "dest": {
    "index": "kibana_sample_data_flights_reindex"
  }
}

# 3.2、多索引

有时候需要将多个索引的数据迁移到一个索引中，比如数据归档，将每天的日志文件归档到当月的归档文件里，source的index可以配置多个索引。

POST _reindex
{
  "source": {
    "index": [
        "kibana_sample_data_flights",
        "kibana_sample_data_flights_001"
      ]
  },
  "dest": {
    "index": "kibana_sample_data_flights_00001"
  }
}
